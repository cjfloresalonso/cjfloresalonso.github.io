recreation

In 2018, I headed a team at a hackathon. My team developed a Brainfuck (BF) self-interpreter from scratch. I was responsible for the design of the interpreter and integrating everyone's code together into the final product. It was a very interesting project, and had a few nifty problems to solve, namely:

  • storing the source code and child programme's tape
  • keeping track of the various places in code
  • actually manipulating data without detroying it
To overcome these problems, I devised a solution involving three kind of seperate sections of memory, each with their own role: one to keep track of what's where in the memory section, the actual memory section containing the child source code and the child tape, and a third section for copying, moving, and operating on values.

The memory section was the easiest, just load the source code symbols to memory, pad it a little just to be sure, and leave the rest blank. The memory management, however, was slightly more complicated. Since it would be a huge waste of memory if the interpreter had a fourth section for source code (once the code is loaded, every fourth byte would be empty), the memory section was split into source code and the child tape. This required various markings: where the different parts started and ended, and where the program was currently pointing to in each of them. This is what occupied the first section, with a little padding just for safety.
The final section was used for comparisons and moving/copying values around. Because BF has no branch/test statements performing conditional logic was done in a unique way: the cell value we were comparing was copied to one of these cells, the value we were checking for was subtracted, the cell was then boolean not-ed, and finally if it were true, the corresponding branch would be executed. This was a huge pain, and the reason we didn't finish in the 36 hours.

All in all a very fun project, with some pretty complex problems to solve - all in my first semester of uni. You can check it out over on my github.

Over the summer I wrote the Nokia classic game Snake in C for a bit of fun. It's a simple little thing, but was was my first time learning to use another library straight from the documentation. The waits around for you to press a button and update the game accordingly, then does it all again. The aim is to use this code as the base for a command line Tron game, which is multiplayer over the internet. It would be pretty cool little project to teach myself threading and sockets.

Head on over to my Github it you want to check it out.
